# v0.5 Character Platform Core

## 1. Stage Position

v0.5 的目标是把产品对象从 `Avatar` 拉回 `Character`。

此前 v0.2-v0.4 已经验证了 TTS、ASR、Live2D、口型、realtime session 等运行时能力。它们继续保留，但从 v0.5 开始只作为 Character 的能力模块存在。

本阶段不追求继续扩展 WebRTC、复杂动作系统或图像生成能力，而是先补齐 AI Character Platform 的产品骨架。

## 2. Goal

v0.5 完成后，系统应该能表达这条主链路：

```text
Character Library
-> Character Profile
-> Character Asset / Appearance
-> Scene Binding
-> Memory Policy
-> Workflow Permission
-> Conversation Runtime with characterId and sceneId
```

必须交付：

- `Character` 成为一等产品对象。
- 角色库入口从 `/avatar` 迁移到 `/characters`，旧入口保留兼容。
- Conversation runtime 明确接收并持久化 `characterId`。
- Scene、Memory、Workflow 有清晰数据模型和第一版 API 边界。
- 旧 Avatar Runtime 作为 Character Appearance Runtime 存在，不再主导产品结构。

## 3. Non-goals

v0.5 不做：

- 不调用 `image_gen`。
- 不接真实图片生成 Provider。
- 不做完整工作流编排器。
- 不做复杂多租户权限。
- 不重写 TTS / ASR / Live2D runtime。
- 不删除旧 `/avatar` 与 `/api/avatar-*` 接口。

## 4. Migration Strategy

### Phase A: Compatibility Layer

保持现有 `AvatarProfile / AvatarAsset / AvatarGenerationJob` 不变，新增 Character 语义层。

```text
CharacterProfile
  -> appearanceProfileId
  -> AvatarProfile
```

这样可以复用已经跑通的 Static / Live2D / VRM 渲染链路。

### Phase B: Runtime Binding

Conversation 增加：

```text
characterId
sceneId
```

发送消息时：

```text
conversation.characterId
-> CharacterProfile
-> Persona + Memory + Scene Context
-> AvatarProfile for appearance runtime
```

### Phase C: UI Rename

- 新主入口：`/characters`
- 旧兼容入口：`/avatar`
- 页面标题：AI 角色管理
- 内部 `Avatar*` 组件名可暂时保留，避免一次性大规模重命名。

### Phase D: API Rename

第一阶段新增兼容 API：

```text
GET /api/characters
POST /api/characters
GET /api/characters/:characterId
PATCH /api/characters/:characterId
```

内部可以先代理或包装现有 `AvatarProfile` 能力。等 CharacterProfile 表稳定后，再逐步迁移。

## 7. Page Scope

### Character Library

第一版页面能力：

- 角色列表。
- 角色基础信息。
- 外观资产绑定。
- 声音配置。
- 默认场景。
- 状态：draft / active / disabled。

### Character Studio

第一版不调用图片生成，只做：

- 手动创建角色。
- 上传参考图或头像。
- 编写角色设定。
- 绑定已有角色资产。

图像生成保留为未来 Provider 能力。

### Scene

第一版支持：

- 知识库助手。
- 主播角色。
- 闲聊陪伴。
- 自定义场景。

### Memory

第一版支持：

- 列表查看。
- 手动新增。
- 禁用。
- 删除。
- 对话摘要写入先做服务端函数，不急于自动化。

### Workflow

第一版支持：

- 静态 workflow 定义。
- 手动运行。
- 执行日志。
- 需要确认的 workflow 不自动执行。

```text

  character-workflow-panel.tsx

  repository.ts
  presenter.ts
  schema.ts
```

不要继续把页面、业务逻辑、API 请求、hooks、types 混在一个大文件里。

## 10. Current Risk

| 风险 | 处理 |
| --- | --- |
| 一次性重命名 Avatar 导致 runtime 破坏 | 先新增 Character 语义层，保留 Avatar runtime |
| 记忆过早自动写入导致脏数据 | 先手动管理，自动写入后置 |
| Workflow 权限失控 | 第一版只允许手动执行和确认 |
| image generation 当前不可用 | 先支持上传图片和手动角色设定 |
| 旧文档继续误导开发 | 12-15 标记为历史运行时路线，18 作为当前 v0.5 主线 |
