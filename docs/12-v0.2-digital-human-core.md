# v0.2 Digital Human Core 阶段文档

## 1. 阶段结论

v0.1 已完成工程骨架和最小闭环，但还不是完整数字人产品。

v0.2 的目标是补齐“数字人核心业务能力”：

```text
用户输入
-> LLM 流式回复
-> 真实 TTS 生成语音
-> Avatar 资产加载
-> 音频播放驱动口型 / 表情 / 状态
-> 可配置或生成虚拟人物
```

WebRTC 不作为 v0.2 第一优先级。原因是当前项目还缺真实 Avatar 渲染、口型同步和人物资产管理，直接上 WebRTC 会把复杂度堆到传输层，而不是补齐数字人核心体验。

## 2. 当前边界

### v0.1 已完成

- Landing / Login / Workspace 页面骨架。
- LLM 流式对话。
- DeepSeek / OpenAI-compatible LLM Provider。
- 会话和消息持久化。
- Mock TTS 音频闭环。
- Static Avatar 状态切换。
- Provider / Avatar / Prompt / Playground / Settings 管理页。

### v0.2 需要补齐

- 真实 TTS Provider。
- Avatar Asset 数据模型。
- 虚拟人物生成或上传。
- 数字人渲染层。
- 音频驱动口型和状态同步。
- 更完整的 Avatar Runtime 事件。

### v0.2 暂不做

- 多租户权限系统。
- 复杂 WebRTC 实时音视频。
- 商业级动捕。
- 完整 Live2D 编辑器。
- 完整 3D 角色编辑器。

## 3. 技术路线选择

### 推荐路线

v0.2 采用“三步走”：

1. **真实语音**：先接入真实 TTS Provider，替换 mock 音频。
2. **可见人物**：实现 Avatar 资产库，支持上传或生成静态数字人图像。
3. **可动人物**：先做音量驱动的简化口型和状态动画，再接 Live2D / VRM。

### Live2D 与 VRM 选择

| 方案 | 优点 | 风险 | v0.2 建议 |
| --- | --- | --- | --- |
| Static + CSS | 实现快，可验证业务闭环 | 表现力弱 | P0 |
| Live2D | 适合 2D 数字人，口型/表情成熟 | 模型资产格式和授权复杂 | P1 |
| VRM + Three.js | 3D 扩展强 | 模型加载、性能、动作系统复杂 | P2 |
| WebRTC | 低延迟实时语音 | 依赖前面所有能力 | v0.3 |

结论：v0.2 先做 **Static Generated Avatar + Real TTS + Lip Sync Runtime**，再预留 Live2D / VRM 插槽。

## 4. v0.2 模块拆分

### M9 Real TTS Provider

目标：让回复能生成真实语音。

交付物：

- `OpenAICompatibleTTSProvider`
- TTS Provider 配置和测试
- 支持 `voice`、`model`、`format`
- `/api/tts` 调用真实 Provider
- Chat Runtime 使用默认 TTS Provider

建议环境变量：

```env
TTS_PROVIDER=mock
DEFAULT_TTS_BASE_URL=
DEFAULT_TTS_API_KEY=
DEFAULT_TTS_MODEL=
DEFAULT_TTS_VOICE=
DEFAULT_TTS_FORMAT=mp3
```

Provider 选择策略：

- 优先实现 OpenAI-compatible TTS，不绑定单一厂商。
- 保留 `custom-http` Provider，用于接入 CosyVoice、Azure、火山、MiniMax 等服务。
- 环境变量 Provider 只作为默认配置，数据库 ProviderConfig 优先级更高。
- Provider 必须统一返回 `audioUrl`、`mimeType`、`durationMs`，后续可选返回 `marks`。
- 不要求所有 Provider 支持 speech marks；无 marks 时用音量分析降级。

验收：

- Provider 管理页可配置 TTS。
- TTS Provider 测试返回真实音频 URL 或可播放音频。
- Chat 回复完成后自动播放真实语音。
- TTS 失败时文本回答仍保留。

### M10 Avatar Asset 数据模型

目标：把“数字人形象”从页面状态变成可管理资产。

建议新增表：

```text
AvatarAsset
- id
- userId
- profileId?
- type: image | live2d | vrm
- name
- status: active | processing | failed | deleted
- source: upload | generated | remote
- storageKey
- publicUrl
- mimeType
- size
- width?
- height?
- metadata
- createdAt
- updatedAt
```

```text
AvatarGenerationJob
- id
- userId
- profileId?
- providerConfigId?
- status: pending | running | completed | failed
- prompt
- negativePrompt?
- style
- resultAssetId?
- errorMessage?
- startedAt?
- completedAt?
- createdAt
- updatedAt
```

存储方案：

| 场景 | 存储位置 | 说明 |
| --- | --- | --- |
| 本地开发 | `storage/avatar-assets` | 直接落盘，便于调试 |
| Preview 部署 | S3 / R2 / OSS | 避免依赖临时文件系统 |
| Production | S3 / R2 / OSS + CDN | 用 `storageKey` 管理真实对象 |

资产表只保存元数据和引用：

- `storageKey`：内部对象路径。
- `publicUrl`：可访问 URL，可为空。
- `metadata`：生成参数、尺寸、seed、Provider 响应摘要。

上传限制：

- 图片类型：`image/png`、`image/jpeg`、`image/webp`。
- 单文件大小：第一版限制 8 MB。
- 图片尺寸：建议最大 2048x2048。
- Live2D / VRM 文件先只做占位，不开放完整上传。
- 删除资产使用软删除，避免误删已绑定 Profile。

验收：

- Avatar 配置页可以选择或预览资产。
- 支持上传静态图片作为数字人形象。
- 资产和 AvatarProfile 解耦。

### M11 虚拟人物生成

目标：通过 Provider 生成数字人头像或半身图。

Provider 能力：

```ts
interface ImageGenerationProvider {
  generate(input: ImageGenerationInput): Promise<ImageGenerationResult>;
}

interface ImageGenerationInput {
  prompt: string;
  negativePrompt?: string;
  style?: string;
  width?: number;
  height?: number;
  signal?: AbortSignal;
}

interface ImageGenerationResult {
  imageUrl?: string;
  imageBytes?: ArrayBuffer;
  mimeType: string;
  seed?: string;
  metadata?: Record<string, unknown>;
}
```

第一版可以先支持：

- 本地上传。
- 远程图片 URL。
- 生成任务接口预留。

真实生成 Provider 可接：

- OpenAI Images
- Replicate
- Stability
- ComfyUI
- 自定义 HTTP

异步任务机制：

- 图片生成必须通过 `AvatarGenerationJob` 管理，不阻塞主请求。
- 第一版可以使用同步执行 + job 状态落库，但 API 形态保持异步。
- 前端使用轮询查询任务状态。
- 后续可替换为队列：BullMQ、Inngest、Trigger.dev、Cloud Tasks。
- Job 失败必须记录 `errorMessage`，并允许重试。

生成安全边界：

- 保存原始 prompt 和 negative prompt。
- 禁止默认生成真实公众人物、未成年人、敏感身份冒用。
- 生成结果必须进入用户确认流程，再绑定 AvatarProfile。
- Provider 返回的远程图片必须下载入自有存储，避免外链失效。

验收：

- 用户能创建一个 AvatarProfile。
- 用户能上传或生成一个头像/半身像。
- 生成结果能绑定到 AvatarProfile。

### M12 Avatar Runtime 与渲染层

目标：把 Avatar 从“状态标签”升级为“可被事件驱动的渲染层”。

Runtime 状态：

```text
idle
listening
thinking
speaking
interrupted
error
```

Runtime 事件：

```text
avatar.asset.loaded
avatar.state.changed
avatar.speech.started
avatar.speech.viseme
avatar.speech.volume
avatar.speech.ended
avatar.motion.play
avatar.expression.set
```

事件载荷：

```ts
type AvatarRuntimeEvent =
  | {
      type: "avatar.asset.loaded";
      assetId: string;
      assetType: "image" | "live2d" | "vrm";
      url: string;
    }
  | {
      type: "avatar.state.changed";
      state: "idle" | "listening" | "thinking" | "speaking" | "interrupted" | "error";
      reason?: string;
    }
  | {
      type: "avatar.speech.started";
      messageId: string;
      audioUrl: string;
      durationMs?: number;
    }
  | {
      type: "avatar.speech.volume";
      messageId: string;
      volume: number;
      mouthOpen: number;
      timeMs: number;
    }
  | {
      type: "avatar.speech.ended";
      messageId: string;
    }
  | {
      type: "avatar.expression.set";
      expression: "neutral" | "happy" | "thinking" | "sad" | "error";
      intensity?: number;
    };
```

第一版渲染策略：

- Static image：图片 + CSS 呼吸动画。
- Speaking：基于音频播放状态做嘴部光效或简单 mouth overlay。
- Thinking：背景光效 / loading pulse。
- Error：降级静态展示。

验收：

- Avatar Stage 能加载 AvatarAsset。
- Chat 事件能驱动 Avatar 状态。
- 音频播放时 Avatar 进入 speaking。
- 音频结束后回到 idle。

### M13 Lip Sync / Audio Sync

目标：让语音和 Avatar 表现同步。

第一版不要做复杂 viseme，可以先用音量分析：

- `AudioContext`
- `AnalyserNode`
- RMS volume
- 映射 mouth openness

后续增强：

- TTS Provider 返回 speech marks。
- viseme 时间轴。
- Live2D 参数映射。
- VRM blendshape 映射。

验收：

- 播放音频时能得到实时音量。
- Avatar mouth openness 随音量变化。
- 无 speech marks 时可用音量驱动降级。

### M14 WebRTC Realtime

目标：低延迟实时语音对话。

放到 v0.3，除非当前产品目标改为实时语音优先。

需要前置能力：

- 真实 ASR。
- 真实 TTS 或 Realtime API。
- Avatar Runtime。
- 音频中断和 barge-in。
- Realtime session 状态。

建议接口：

```text
POST /api/realtime/session
POST /api/realtime/token
WS   /api/realtime/events
```

验收：

- 用户可开启实时会话。
- 麦克风输入实时转写。
- LLM 可中断。
- Avatar 可同步 listening / thinking / speaking。

## 5. v0.2 Demo 验收场景

完整 Demo 必须能按以下路径走通：

1. 用户打开 `/avatar`。
2. 创建或选择一个 AvatarProfile。
3. 上传一张数字人形象图片，或发起一次生成任务。
4. 资产生成或上传成功后，绑定到 AvatarProfile。
5. 用户打开 `/models`，配置真实 TTS Provider。
6. Provider 测试返回可播放音频。
7. 用户打开 `/conversation`。
8. 页面加载绑定的 AvatarAsset。
9. 用户发送文本问题。
10. LLM 流式返回文本。
11. TTS 生成真实语音。
12. 音频播放时 Avatar 进入 `speaking`。
13. mouth openness 随音频音量变化。
14. 音频结束后 Avatar 回到 `idle`。
15. 会话、消息、audioUrl、AvatarProfile 关联信息可持久化。

Demo 验收失败条件：

- 只能展示静态卡片，不能加载用户资产。
- TTS 仍只返回 mock silence。
- 音频播放和 Avatar 状态没有联动。
- 生成任务失败后没有错误状态。
- 上传资产无法复用或无法绑定 Profile。

## 8. 成本、限流与安全

### 成本限制

| 能力 | v0.2 默认限制 |
| --- | --- |
| TTS | 单次最多 2,000 字符 |
| TTS Provider 测试 | 每次最多 200 字符 |
| 图片上传 | 单文件最多 8 MB |
| 图片生成 | 单用户每小时最多 20 次 |
| Avatar assets | 单用户最多 100 个 active assets |

### 超时策略

| 操作 | 超时 |
| --- | --- |
| TTS synthesize | 30 秒 |
| 图片上传处理 | 15 秒 |
| 图片生成 job | 120 秒 |
| Provider test | 20 秒 |

### 安全要求

- Provider API Key 只允许服务端读取。
- 上传文件必须检查 MIME type 和文件大小。
- 不信任客户端传入的 `publicUrl`。
- 生成图片任务必须保存 prompt，便于排查和审计。
- 删除资产必须检查归属用户。
- 软删除资产时不能破坏历史会话。
- 前端不能直接访问未授权 storage key。

### 降级策略

| 能力 | 失败时降级 |
| --- | --- |
| Real TTS | 保留文本，显示 TTS failed，允许重试 |
| Avatar asset | 使用默认静态 Avatar |
| Image generation | 保留 failed job，可重试 |
| Lip sync | 只做 speaking 状态，不显示 mouth |
| Storage public URL | 走 API proxy 读取 |

- `POST /api/avatar-generation-jobs` 创建 job。

### 集成 smoke

- 上传图片 -> 创建资产 -> 绑定 AvatarProfile。
- 配置 TTS Provider -> 测试成功。
- Chat -> LLM -> TTS -> audioUrl 持久化。
- Conversation 页面加载 AvatarAsset。
- 音频播放时 Avatar 进入 speaking。

### 手动验收

## 12. v0.2 完成定义

v0.2 完成时必须满足：

- 用户能配置真实 TTS Provider。
- Chat 回复能播放真实 TTS 音频。
- 用户能上传或生成一个数字人形象资产。
- AvatarProfile 能绑定资产。
- Conversation 页面能展示绑定的数字人形象。

- 失败时能降级到静态 Avatar 和文本回复。
- WebRTC 有设计文档和接口边界，但不要求实现。
